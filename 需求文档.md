# BoardGameSimulator（一起来玩桌游吧！）- 需求文档

> 版本：v1.0 | 状态：需求冻结，进入开发准备阶段

---

## 1. 项目概述

### 1.1 一句话定位
一个**去中心化的通用桌游模拟器**——主持人的手机负责逻辑运算和状态广播，所有玩家的手机仅作为显示器与触控输入器，通过P2P直连实现零服务器多人游戏。

### 1.2 项目性质
个人兴趣项目，“做着玩”。不以商业化为目标，但代码质量按可维护、可扩展的标准要求。

---

## 2. 核心目标与边界

### 2.1 做什么（边界内）
- **游戏类型**：卡牌 + 骰子 + 棋子 + 回合制的桌游（覆盖约70%的常见桌游类型）
- **交互模式**：回合制 / 非实时（允许1-2秒网络延迟）
- **通信模式**：P2P直连（WebRTC），零自有服务器
- **平台**：Web应用 + PWA（可添加到主屏幕，全屏运行）
- **编辑器**：提供可视化编辑器，通过配置JSON生成游戏规则，无需手写代码（L3除外）

### 2.2 不做什么（边界外）
- ❌ 实时动作类游戏（如《德国心脏病》抢铃机制）
- ❌ 需要物理引擎的桌游（如《弹指赛车》）
- ❌ 自由涂鸦/画板同步（笔迹数据量过大，不适合P2P）
- ❌ 超复杂嵌套连锁规则（如《游戏王》无限连锁，这类逻辑必须走L3脚本）
- ❌ 任何形式的自有后端服务器（WebSocket / HTTP API）
- ❌ APK / IPA 打包，不上架任何应用商店
- ❌ 用户账号系统、云端存档、匹配系统

### 2.3 MVP测试对象
**《斗地主》**（标准规则，无变种）
- 固定3人（1地主 + 2农民）
- 54张牌（含大小王），每人17张手牌，3张底牌
- 支持叫地主、出牌（含牌型校验）、不出、胜负判定（农民协作胜利）

## 4. 核心架构设计

### 4.1 三层抽象标准（L1 / L2 / L3）

| 层级 | 命名 | 职责 | 存储格式 | 编辑方式 |
| :--- | :--- | :--- | :--- | :--- |
| **L1** | 数据层 | 定义"有什么"：卡牌列表、牌堆构成、骰子面数、棋盘格名、初始资源、UI布局映射 | JSON（Schema严格约束） | 可视化编辑器（表单填表） |
| **L2** | 行为层 | 定义"怎么动"：触发器 + 条件 + 原子动作序列（抽牌、出牌、移动、计分等） | JSON（固定Action枚举） | 可视化编辑器（卡片序列 + 下拉参数 + 拖拽排序） |
| **L3** | 逻辑层 | 处理L2无法覆盖的复杂逻辑：牌型识别、特殊结算、跨回合记忆、自定义AI | JavaScript函数 | 编辑器内嵌CodeMirror代码框 |

**核心原则**：
- L1 + L2 必须**完全由编辑器生成**，禁止手写JSON（防格式错误）
- L3 为可选项，没有L3时游戏也能运行（斗地主必须包含L3）
- L3 不覆盖L2，而是**注入钩子**（Hook），在L2执行前后或特定节点被调用

#### 运行时校验（engine.loadGame）

引擎加载 `game.json` 时必须执行以下校验，校验失败返回结构化错误而非崩溃：

1. **字段完整性**：L1必填字段（卡牌列表、牌堆构成、玩家数等）不可缺失
2. **L3语法检查**：尝试 `new Function(script)` 语法解析，失败则标记为"L3加载失败"但不阻止游戏启动（L3为可选）
3. **引用完整性**：L2中引用的 Action/Condition/Component 名称必须在对应Registry中已注册
4. **编辑器预检**：导出的 `game.json` 必须通过编辑器内置的 `exporter.validate()` 校验

校验错误格式：
```typescript
interface ValidationError {
  level: 'error' | 'warning';
  path: string;      // 字段路径，如 "l1.cards[3].value"
  message: string;   // 中文错误描述
}
```

### 4.2 网络架构（主机权威模式）

```
┌─────────────┐      动作(Action)       ┌─────────────┐
│  玩家设备   │ ────────────────────────> │  主持人设备  │
│  (仅渲染)   │                           │  (权威主机)  │
│             │ <──────────────────────── │             │
└─────────────┘     状态快照(Snapshot)    └─────────────┘
                                               │
                                               ▼
                                         ┌─────────────┐
                                         │  其他玩家   │
                                         │  (同左)     │
                                         └─────────────┘
```

**规则**：
- 所有游戏逻辑仅在主持人设备上运行
- 玩家设备发送"动作意图" → 主机验证并执行 → 广播完整状态快照
- 冲突处理：后到达的Action包直接丢弃，客户端收到最新状态后强制覆盖
- 同步粒度：以"动作"为单位，一个动作结束后广播一次

#### 4.2.1 玩家视图过滤（PlayerView）

主机维护完整 `GameState`（含所有玩家手牌），但广播时按玩家生成过滤后的 `PlayerView`：

```typescript
interface PlayerView {
  version: number;          // 复用GameState的版本号
  playerIndex: number;      // 目标玩家索引
  phase: GamePhase;
  currentTurn: number;
  winner: number | null;
  // 以下字段经过过滤（见7.4信息可见性配置）
  players: PlayerViewData[];
  publicState: PublicState; // 所有玩家均可见的公共信息
}
```

**定向发送**：主机通过各玩家的独立 DataChannel 仅发送该玩家的 `PlayerView`，替代"广播同一份快照"。

### 4.3 扩展机制（插件化架构）

核心引擎不硬编码任何游戏特定逻辑，通过**注册表模式**支持扩展：

#### 4.3.1 动作扩展（Action Registry）
```typescript
interface IActionHandler {
  execute(state: GameState, params: any, context: ActionContext): GameState;
  validate?(state: GameState, params: any, context: ActionContext): boolean;
}

// 全局注册表，引擎动态读取
ActionRegistry.set('action_name', { execute, validate });
```

#### 4.3.2 条件扩展（Condition Registry）
```typescript
interface IConditionHandler {
  check(state: GameState, params: any, context: ActionContext): boolean;
}

ConditionRegistry.set('condition_name', { check });
```

#### 4.3.3 UI组件扩展（Component Registry）
```typescript
interface UIComponent {
  render(data: any, dispatch: (action: string, payload: any) => void): HTMLElement;
  update?(element: HTMLElement, newData: any): void;
}

ComponentRegistry.set('component_name', { render, update });
```

#### 4.3.3.1 UI布局Schema（ui_layout）

```typescript
interface UILayout {
  slots: {
    [slotName: string]: {
      component: string;   // ComponentRegistry中注册的组件名
      params?: Record<string, any>;  // 传递给组件的静态参数
    }
  };
  // 预定义插槽名（根据5.1.2固定模板）
  presetSlots: ['top_bar', 'main_area', 'bottom_bar'];
}
```

**示例（斗地主）**：
```json
{
  "slots": {
    "top_bar":     { "component": "info_area" },
    "main_area":   { "component": "play_area" },
    "bottom_bar":  { "component": "hand_area", "params": { "maxSelect": 20 } }
  }
}
```
插槽命名规范：蛇形命名（snake_case），与DOM容器id一一对应。

#### 4.3.4 L3脚本事件监听（非覆盖式）
```javascript
// L3脚本中可注册多个监听器
game.on('before_action', (state, action) => { /* 前置钩子 */ });
game.on('after_state_update', (state) => { /* 后置钩子 */ });
```

#### 4.3.5 L3同步查询接口（FunctionRegistry）

除异步事件Hook外，L3还可注册**同步查询函数**，引擎在需要时直接调用并等待返回值：

```typescript
interface IL3Function {
  (state: GameState, ...args: any[]): any;
}

// L3脚本中注册
FunctionRegistry.set('checkPattern', (state, cards) => { /* 返回牌型对象 */ });
FunctionRegistry.set('getValidPlays', (state, hand, lastPlay) => { /* 返回可选牌组列表 */ });
```

引擎调用方式：
```typescript
const pattern = FunctionRegistry.get('checkPattern')(state, selectedCards);
```

与事件Hook的区别：事件Hook是异步通知（fire-and-forget），函数查询是同步调用（需要返回值）。斗地主的牌型识别函数（6.3）均通过FunctionRegistry暴露给引擎。

#### 4.3.6 L3沙箱执行（Web Worker）

所有L3脚本（事件Hook + 同步查询函数）在**独立Web Worker**中执行，Worker不具备以下权限：
- 无DOM访问（`document`、`window`均为undefined）
- 无网络访问（`fetch`、`WebSocket`、`XMLHttpRequest`通过CSP `connect-src 'none'`阻断）
- 无存储访问（`localStorage`、`sessionStorage`、`IndexedDB`不可用）

引擎与Worker通过 `postMessage` + `MessageChannel` 通信：
```
引擎主线程                  L3 Worker
   │                          │
   │── { type: 'hook',       │
   │     name: 'before_action',│
   │     state, action } ──→  │
   │                          │── 执行L3回调
   │   { result: modified } ←─│
   │                          │
   │── { type: 'query',      │
   │     name: 'checkPattern',│
   │     cards } ──────────→  │
   │                          │── 执行FunctionRegistry函数
   │   { result: pattern } ←──│
```

Worker 打包方式：`new Worker(new URL('./l3.worker.ts', import.meta.url), { type: 'module' })`（Vite原生支持）。

## 5. 功能需求详述

### 5.1 游戏客户端（主持人/玩家共用）

#### 5.1.1 屏幕适配
- **设计基准**：宽度 400px，高度随视口等比缩放
- **适配方式**：CSS `transform: scale()` 全局缩放填满全屏
- **不做**：任何 `@media` 流式折行适配
- **方向**：通过 `manifest.json` + `screen.orientation.lock()` 组合策略软锁定竖屏/横屏

#### 5.1.2 UI布局规则（固定模板）
```
+---------------------------+
|  顶部信息栏（回合/分数/状态） |
+---------------------------+
|                           |
|   中央主区域               |
|   （棋盘 / 牌桌 / 出牌区）  |
|                           |
+---------------------------+
|  底部操作区（按钮/手牌列表） |
+---------------------------+
```

#### 5.1.3 核心交互流程
1. **主持人**：打开页面 → 点击"创建房间" → 导入或选择游戏配置 → 生成房间码/二维码
2. **玩家**：扫码或输入房间码 → 加入房间 → 等待主持人开局
3. **对局中**：
   - 主持人负责所有逻辑运算，界面显示完整游戏状态
   - 玩家界面仅显示该玩家可见信息（如手牌），操作按钮触发Action发送给主机
   - 玩家操作后等待主机回传最新状态快照，UI刷新

### 5.2 可视化编辑器（PC端，/editor 路由）

#### 5.2.1 整体布局（三栏式）
```
+------------------+-----------------------------------+------------------+
|  左侧：结构树     |   中间：编辑区（动态变化）         |  右侧：属性面板   |
|  (卡牌库/棋盘/    |   (根据左侧选中项渲染不同表单)      |  (参数配置)      |
|   规则列表)       |                                    |                  |
+------------------+-----------------------------------+------------------+
|                 底部：JSON预览 / 导出 / 试玩（控制台日志）              |
+-----------------------------------------------------------------+
```

#### 5.2.2 L1编辑（数据层）
- **卡牌管理**：表格形式批量编辑（名称、花色、点数、扩展属性）
- **棋盘管理**：选择类型（网格/环状/线性），配置格子数量与标签
- **玩家资源**：配置初始HP、金币、手牌上限等
- **UI布局映射**：`ui_layout` 字段配置各插槽对应的组件名

#### 5.2.3 L2编辑（行为层）
- **规则列表**：左侧树形菜单展示所有规则（按触发器分组）
- **动作序列**：中间区域为纵向卡片列表，支持拖拽排序（SortableJS）
- **动作卡片**：下拉选择动作类型 → 自动展开对应参数输入框
- **条件块**：支持单层 `if` 条件嵌套（折叠显示）

#### 5.2.4 L3编辑（逻辑层）
- **代码框**：集成 CodeMirror，JavaScript语法高亮
- **上下文注入**：自动注入 `__context__` 对象（state / currentPlayer / utils）
- **事件API**：提供 `game.on()` / `game.off()` 接口

#### 5.2.5 导出与预览
- **导出**：点击"导出"按钮，自动下载 `game.json` 单文件
- **试玩**：点击"试玩"，在控制台打印每一步执行的动作和状态变化（无动画）

### 5.3 P2P通信层

#### 5.3.1 信令流程
1. 主持人调用 `createRoom()` → 生成房间码
2. 玩家调用 `joinRoom(code)` → 通过公共Nostr Relay完成WebRTC信令交换
3. 所有玩家与主持人建立P2P直连（星型拓扑）

#### 5.3.2 数据通道
- 使用 WebRTC DataChannel（可靠传输模式）
- 消息格式：JSON序列化
- 消息类型：`action`（玩家操作）/ `state`（主机广播状态）/ `sync`（握手确认）

#### 5.3.3 断线处理（MVP边界）
- 主持人断线 → 房间解散，所有玩家收到提示
- 玩家断线 → 主机标记该玩家为"掉线"，游戏继续（MVP阶段不做重连）
- **预留接口**：主机每回合将 `GameState` 快照写入 localStorage（key: `room_snapshot_{roomCode}`），供未来 Phase 实现主机迁移/重连时读取

## 6. 斗地主专用需求（MVP验证目标）

### 6.1 L1数据定义
- 54张牌（4花色×13张 + 大小王）
- 初始发牌：每人17张，底牌3张
- 玩家数：固定3人
- 初始状态：所有玩家未叫地主

### 6.2 L2规则清单
| 触发器 | 条件 | 动作序列 |
| :--- | :--- | :--- |
| `on_game_start` | 无 | 发牌(17张/人) → 亮底牌 → 进入叫地主阶段 |
| `on_call_phase` | 无 | 按顺序叫地主（可叫/不叫） |
| `on_landlord_chosen` | 有人叫地主 | 地主获得底牌 → 进入出牌阶段 |
| `on_turn_start` | 当前玩家未出完牌 | 等待玩家出牌或不出 |
| `on_card_played` | 牌型合法且 > 上家 | 更新出牌区 → 检查是否出完 → 进入下家回合 |
| `on_pass` | 无 | 进入下家回合 |
| `on_cards_empty` | 任意玩家手牌为0 | 判定胜利（地主/农民） |

### 6.3 L3必需函数（牌型识别）
| 函数名 | 职责 |
| :--- | :--- |
| `checkPattern(cards)` | 识别牌型（单张/对子/三张/顺子/连对/飞机/炸弹/火箭/三带一/三带二/四带二） |
| `comparePatterns(patternA, patternB)` | 比较两组牌的大小（同一牌型时比较点数） |
| `getValidPlays(hand, lastPlay)` | 给定手牌和上家出的牌，返回所有合法的出牌组合 |

### 6.4 UI组件需求
| 组件名 | 位置 | 功能 |
| :--- | :--- | :--- |
| `hand_area` | 底部 | 展示当前玩家手牌，点击选中 |
| `play_area` | 中央 | 显示上家/下家/当前玩家的出牌 |
| `landlord_call_area` | 中央（临时） | 叫地主/不叫按钮 |
| `info_area` | 顶部 | 显示当前回合、地主标识、剩余牌数 |
| `pass_button` | 底部 | 不出按钮（当上家出牌后可点击） |

## 7. 数据流与状态管理

### 7.1 状态树结构（GameState）
```typescript
interface GameState {
  version: number;           // 状态版本号，用于冲突检测
  players: PlayerState[];    // 所有玩家状态
  deck: Card[];             // 牌堆
  discard: Card[];          // 弃牌堆（斗地主中为出牌区）
  bottomCards: Card[];      // 底牌（3张）
  landlordIndex: number;    // 地主索引（-1表示未确定）
  currentTurn: number;      // 当前回合玩家索引
  phase: 'idle' | 'calling' | 'playing' | 'ended';
  lastPlay: PlayInfo | null; // 上家出的牌
  passCount: number;        // 连续不出次数
  winner: number | null;    // 胜利方（地主索引 或 农民队伍标识）
}
```

### 7.2 Action结构
```typescript
interface GameAction {
  type: 'call_landlord' | 'play_cards' | 'pass' | 'start_game';
  playerIndex: number;
  payload?: any;            // 如出牌时携带 selectedCards
  timestamp: number;
}
```

### 7.3 状态更新流程
1. 玩家设备构造 `GameAction` → 发送给主机
2. 主机接收 → 调用 `reducer(state, action)` → 返回新状态
3. 主机校验（如牌型合法性）→ 通过则应用新状态，拒绝则返回错误 `{ code: string, message: string }` 给**发起操作的玩家**
4. 主机对每个玩家调用 `buildPlayerView(state, playerIndex)` 生成过滤后的 `PlayerView`
5. 主机通过各玩家的独立 DataChannel **定向发送**各自的 `PlayerView`（替代广播完整GameState）
6. 所有玩家设备用收到的 `PlayerView` 覆盖本地视图 → 重新渲染UI

### 7.4 信息可见性配置（L1声明式定义）

每个游戏在 L1 配置的 `visibility` 字段中声明各状态字段的可见性粒度：

```typescript
interface VisibilityConfig {
  [fieldPath: string]: {
    mode: 'full' | 'count' | 'hidden' | 'owner_only';
    description: string;
  };
}
```

**可见性模式**：
| 模式 | 含义 | 示例 |
| :--- | :--- | :--- |
| `full` | 所有玩家可见完整数据 | 出牌区、当前回合玩家 |
| `count` | 仅可见数量（如手牌数） | `{ count: 17 }` |
| `hidden` | 完全不可见 | 牌堆剩余牌 |
| `owner_only` | 仅该数据所属玩家可见 | 自身手牌 |

**斗地主 visibility 配置示例**：
```json
{
  "players[*].hand":    { "mode": "owner_only", "description": "手牌仅本人可见" },
  "players[*].handCount": { "mode": "full", "description": "所有玩家可见手牌数量" },
  "bottomCards":        { "mode": "hidden", "description": "叫地主前底牌不可见" },
  "deck":               { "mode": "hidden", "description": "发牌后牌堆无意义" },
  "discard":            { "mode": "full", "description": "出牌区所有人可见" }
}
```

`buildPlayerView()` 函数根据此配置自动过滤 `GameState` → `PlayerView`。

### 7.5 错误处理与边界情况

**错误消息格式**（统一为JSON）：
```typescript
interface ErrorResponse {
  code: string;    // 如 "INVALID_PATTERN" "NOT_YOUR_TURN" "GAME_NOT_STARTED"
  message: string; // 中文描述，如 "不是你的回合"
}
```

**错误处理规则**：
1. 校验失败时错误仅发送给**发起操作的玩家**（不广播）
2. 客户端收到错误后：弹toast提示（3秒自动消失）+ 操作按钮保持当前状态
3. 网络超时策略：客户端发送Action后3秒未收到响应 → 标记为网络异常 → 显示"连接中..."提示

**边界情况定义**（斗地主示例）：
| 场景 | 错误码 | 行为 |
| :--- | :--- | :--- |
| 非当前回合玩家出牌 | `NOT_YOUR_TURN` | 拒绝，返回错误 |
| 叫地主阶段出牌 | `WRONG_PHASE` | 拒绝，返回错误 |
| 第4人尝试加入满员房间 | `ROOM_FULL` | 拒绝加入 |
| 出牌牌型不合法 | `INVALID_PATTERN` | 拒绝，返回具体原因 |
| 出的牌未在手牌中 | `CARDS_NOT_IN_HAND` | 拒绝 |
| DataChannel断开（玩家侧） | `DISCONNECTED` | UI显示"连接断开"，等待重连 |
| DataChannel断开（主机侧） | — | 标记玩家掉线，游戏继续 |

## 11. 风险与约束

| 风险 | 缓解措施 |
| :--- | :--- |
| WebRTC信令在部分网络环境下失败 | 配置多个Nostr Relay备用节点；降级方案：使用Trystero的MQTT信令 |
| iOS Safari不支持 `screen.orientation.lock()` | 通过PWA的manifest配置实现方向锁定；无法锁定则显示提示 |
| L3脚本安全问题 | L3脚本在Web Worker沙箱中执行，Worker无DOM/network访问权限，通过postMessage与引擎通信（详见4.3.4） |
| 主持人设备性能不足 | 桌游逻辑计算量小，现代手机可轻松应对；限制同时在线≤6人 |
| 状态快照过大 / 信息泄露 | 引入PlayerView机制（详见4.2.1 + 7.4），主机维护完整GameState但向玩家发送过滤后的视图；卡牌数据不含base64图片；单次PlayerView体积约1-3KB，远低于DataChannel MTU（16KB）；复杂游戏上限估算约5-10KB |
